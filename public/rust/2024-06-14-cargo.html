01介绍

一个包管理工具

基本概念

Crate 包. Package 项目

创建一个新的二进制项目. (二进制意味着该项目可以作为一个服务运行或被编译成可执行文件运行)

cargo new hello_world

等价

cargo new hello_world --bin

(--bin 默认参数)

新建的项目结构

  • Cargo.toml 和 Cargo.lock 保存在 package 根目录下
  • 源代码放在 src 目录下
  • 默认的 lib 包根是 src/lib.rs
  • 默认的二进制包根是 src/main.rs
  • 其它二进制包根放在 src/bin/ 目录下
  • 基准测试 benchmark 放在 benches 目录下
  • 示例代码放在 examples 目录下
  • 集成测试代码放在 tests 目录下

常用操作

编译 运行

编译构建 生成可执行的二进制文件 --release是生产模式（大量优化）

cargo build
cargo build --release

再运行

./target/debug/hello_world

编译+运行

cargo run
cargo run --release

指定目标平台 输出会放在 target/XXX/ 目录下

cargo build --target XXX

不指定 则会放在target/下(根据宿主架构)

target下的每个子目录都包含了发布配置的profile

目录描述
target/debug/包含了 dev profile 的构建输出(cargo build 或 cargo build --debug)
target/release/release profile 的构建输出，cargo build --release
target/foo/自定义 foo profile 的构建输出，cargo build --profile=foo

出于历史原因:

  • dev 和 test profile 的构建结果都存放在 debug 目录下
  • release 和 bench profile 则存放在 release 目录下
  • 用户定义的 profile 存在同名的目录下

build后除了执行未见还有 依赖信息文件.d 提供给外部用(说明构建编译成果所需的所有依赖包)

添加/更新依赖

crates.io 是rust社区维护的注册中心，默认从这里下载依赖

手动指定 build时会重新构建

[dependencies]
time = "0.1.12"

命令添加

cargo add time

以上操作 会更新Cargo.lock 锁住对应版本 防止下次build版本不同

更新依赖

cargo update # 更新所有

cargo update -p regex # 更新regex

测试和CI

cargo test：寻找src/下单元测试 tests/下集成测试

Workspace

一个工作空间是由多个Package组成 共享同一个 Cargo.lock、 输出目录target和一些设置

只有工作空间根目录的 Cargo.toml 才能包含 [patch], [replace] 和 [profile.*]，而成员的 Cargo.toml 中的相应部分将被自动忽略

工作空间有两种类型：root package 和虚拟清单( virtual manifest )。

root package

一个package的Cargo.toml包含了[package] 的同时又包含了 [workspace]

举个例子，大名鼎鼎的 ripgrep 就在最外层的 package 中定义了 [workspace] :

[workspace]
members = [
  "crates/globset",
  "crates/grep",
  "crates/cli",
  "crates/matcher",
  "crates/pcre2",
  "crates/printer",
  "crates/regex",
  "crates/searcher",
  "crates/ignore",
]

虚拟清单

一个 Cargo.toml 只有[workspace]

对于没有主 package 的场景或你希望将所有的 package 组织在单独的目录中时 使用

\[workspace\]

如果本地依赖通过path引入 且在工作空间中 则该包自动成为工作空间成员

剩余的成员需要通过 workspace.members 指定 包含了各个成员所在的目录(成员目录中包含了 Cargo.toml ) crates/* 匹配crates 目录下的所有包 exclude 排除 crates/foo foo不算

[workspace]
members = ["member1", "path/to/member2", "crates/*"]
exclude = ["crates/foo", "path/to/other"]

你也可以将一个空的 [workspace] 直接联合 [package] 使用，例如：

[package]
name = "hello"
version = "0.1.0"

[workspace]

此时的工作空间的成员包含:

  • 根 package : "hello"
  • 所有通过 path 引入的本地依赖(位于工作空间目录下)

选择工作空间

cargo会自动查找带有[workspace] 的Cargo.toml

也可以给成员包指定工作空间 package.workspace （适用于成员不在工作空间的子目录下）

选择package

-p 、 --package 命令行参数来指定想要操作的 package --workspace  所有成员

没有任何参数 将使用当前工作目录的中的 package 实体清单型则等价于加了--workspace或者添加default-members

[workspace]
members = ["path/to/member1", "path/to/member2", "path/to/member3/*"]
default-members = ["path/to/member2", "path/to/member3/foo"]

workspace.metadata

与 package.metadata 非常类似，workspace.metadata 会被 Cargo 自动忽略 存一些元信息

[workspace]
members = ["member1", "member2"]

[workspace.metadata.webcontents]
root = "path/to/webproject"
tool = ["npm", "run", "build"]
# ...

缓存

cargo使用了缓存 提升构建效率 已下载的依赖在 CARGO_HOME目录

cargo home

默认是在 $HOME/.cargo/

$ echo $HOME/.cargo/
/Users/sunfei/.cargo/

该目录内容解释:link

在 CI 时缓存 Cargo Home

为了避免持续集成时重复下载所有的包依赖，我们可以将 $CARGO_HOME 目录进行缓存，但缓存整个目录是效率低下的，原因是源文件可能会被缓存两次。

例如我们依赖一个包 serde 1.0.92，如果将整个 $CACHE_HOME 目录缓存，那么serde 的源文件就会被缓存两次：在 registry/cache 中的 serde-1.0.92.crate 以及 registry/src 下被解压缩的 .rs 文件。

因此，在 CI 构建时，出于效率的考虑，我们仅应该缓存以下目录:

  • bin/
  • registry/index/
  • registry/cache/
  • git/db/
