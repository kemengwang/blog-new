Blog New DOH 发布链路复盘

从旧 Next.js 博客，到静态文章、动态索引、Docker SSR、Nginx 分发和一条命令发布。

DATE · 2026-07-02 CATEGORY · blog STATUS · SHIPPED
TL;DR
今天完成了 blog-new 的核心改造：文章页变成单文件静态 HTML，首页和分类页由 Vite + React SSR 服务实时扫描目录生成，Nginx 负责路径分发，Docker 承载 SSR 服务，旧博客 39 篇文章迁移到新样式，并新增 article-to-html-doh Skill，实现 doh --c category 直接生成并推送文章到服务器。

01目标变化

最初的问题不是“怎么再搭一个博客”，而是“怎么让 AI 总结后的内容尽快出现在网站上”。旧仓库是一个较重的 Next.js 博客，发布一篇文章需要走 GitHub push、CI、Docker 构建和部署，这对高频写作太慢。

新的目标被收敛成四点：不做数据库，不做 CMS，不依赖 GitHub CI 发布文章，不把文章发布变成前端工程流程。最终选择的是：AI 生成完整 HTML，上传到服务器静态目录，Nginx 直接托管文章文件。

DECISION
文章页不是应用页面，而是可独立访问、可直接备份、可由任何 AI 客户端生成的单文件 HTML。

02仓库从静态发布器变成动态索引服务

第一版 blog-new 是零依赖静态发布器，使用 npm run new 生成文章、维护 posts.json、重建首页，再用 rsync 发布。但这仍然意味着发布文章要执行 npm 命令，也意味着首页需要被重新生成和上传。

后来调整为更符合目标的结构：Skill 只上传单篇文章 HTML，首页不发布。首页和分类页改成运行时扫描文件系统，由 Node SSR 动态渲染。

阶段方式问题结果
初版npm run new + posts.json仍然像工程发布废弃为主链路
目标版Skill 上传单篇 HTML首页需要自动发现文章引入 SSR 索引服务
当前版静态文章 + 动态索引需要常驻服务用 Docker 跑 Node SSR

03最终路径约定

博客作为主站下的 /blog/ 模块存在，根路径留给未来主站或其他业务。文章路径采用一级目录表达分类，文件名前缀表达日期。

/blog/                                  -> SSR 首页
/blog/<category>/                       -> SSR 分类页
/blog/<category>/<YYYY-MM-DD-slug>.html -> Nginx 静态文章

这个规则让 Skill 不需要维护索引文件，也不需要知道首页如何生成。它只需要稳定地产出一个 HTML 文件，并上传到正确目录。

04Vite + React SSR 首页

首页技术栈选择了 Vite + React SSR，而不是回到 Next.js。原因是这里需要的不是完整 Web App 框架，而是一个很薄的索引渲染器：扫描目录、读取 HTML 元信息、输出首页和分类页。

SSR 服务扫描 BLOG_CONTENT_DIR，跳过 index.html，从每个 HTML 文件中读取 <title>、<meta name="description">、路径分类和日期，然后渲染列表。

STATIC
文章页
Nginx 直接返回
每篇文章是完整 HTML，不依赖 Node 服务渲染。
SSR
首页 / 分类页
Docker 内 Node 服务
访问时扫描文章目录，实时生成索引。
SKILL
发布入口
AI 客户端触发
生成 HTML 后直接上传到服务器文章目录。

05旧博客迁移

旧仓库的文章集中在 data/blog/YYYY/*.md。迁移脚本读取 frontmatter，跳过 draft: true，使用第一个 tag 作为分类目录，把 Markdown 转成 paper proposal 风格的单文件 HTML。

迁移结果是 39 篇已发布文章进入新目录，3 篇草稿被跳过。旧文章里的本地图片被内联为 data:image，这样新站点不再依赖旧仓库的静态资源目录。

npm run migrate:old
# Migrated 39 posts, skipped 3.

这一步之后，首页 SSR 能扫描到 40 篇文章：39 篇旧文加上最早的 Blog New 启动记录。

06服务器部署

服务器是新买的 Ubuntu 26.04 64 位云主机。部署方式是 Nginx 做总入口，Docker 只跑 blog-index 这个 SSR 服务。静态文章目录在宿主机上，容器通过只读 volume 挂载。

/opt/blog-new  -> 项目代码和 Docker 配置
/var/www/blog  -> 文章 HTML 静态目录
127.0.0.1:3436 -> Docker 中的 SSR 服务

Nginx 分发规则如下：/blog/*.html 直接走静态文件，/blog/ 和 /blog/<category>/ 反向代理到 Node SSR。

location ~ ^/blog/(.+.html)$ {
  alias /var/www/blog/$1;
  default_type text/html;
}

location /blog/ {
  proxy_pass http://127.0.0.1:3436;
}
DEPLOY NOTE
服务器在国内，Docker Hub 拉镜像容易超时，因此 Docker 配置了 registry mirrors。远端 npm 在容器构建中出现过 npm 自身异常，最终改成“本地构建 dist + 容器只运行”的模式。

07Docker 构建策略调整

最初 Dockerfile 尝试在服务器构建阶段执行 npm ci 和 npm run build。实际部署时遇到 npm 的 Exit handler never called 异常，导致构建失败。

最后的处理是让本地机器负责构建 dist/server，并把生产运行所需的 react、react-dom、scheduler 一起同步到服务器构建上下文。生产容器不再访问 npm registry，只复制本地构建产物并启动 Node 服务。

这让部署更确定：服务器只负责运行镜像，不负责解析前端依赖。

08DOH Skill

最后把文章生成和发布封装成新的 Codex Skill：article-to-html-doh。它复用原 article-to-html 的 paper proposal 设计系统，但触发规则被严格限制为请求中包含 /doh 或独立 doh。

分类由用户请求里的 --c <category> 指定。例如这篇文章的触发方式就是：

doh --c blog 总结下今天的流程发一篇详细的blog吧

发布脚本会从 HTML 的标题和日期生成文件名，上传到：

/var/www/blog/<category>/<YYYY-MM-DD-slug>.html

真实服务器配置放在 Skill 自己的私有 .env，仓库里只保留 .env.example。这样 Skill 可复用，敏感信息不进入 GitHub。

09当前发布链路

现在发布文章的链路已经压缩到一条意图明确的 AI 指令：

doh --c blog 写一篇文章...
  1. AI 生成文章。 使用 paper proposal 模板，输出完整 HTML。
  2. Skill 解析分类。 优先读取 --c 参数。
  3. 脚本上传文件。 通过 rsync 把 HTML 上传到服务器目录。
  4. Nginx 立即可见。 文章页不经过构建、不经过首页发布。
  5. SSR 自动索引。 首页和分类页下一次访问时自动扫描到新文章。

10后续备份策略

因为文章最终直接写入服务器，如果服务器磁盘出问题，理论上可能丢文章。已确认的备份方向是让服务器每天把 /var/www/blog 同步回 GitHub 主仓库的 public/，作为兜底备份。

这不影响日常发布速度：平时 Skill 只管上传单篇 HTML，定时任务在后台做备份。
