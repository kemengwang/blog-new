把 hanami.top 接到中间层代理：从 403 到 HTTPS

一次小型上线记录：中间层 nginx、上游 Host、证书和跳转重写。

STATUS · PUBLISHED DATE · 2026-07-03 AUTHOR · Hanami Ops
TL;DR
今天把 hanami.top 接到一台中间层服务器上，由它用 nginx 反向代理到真实博客服务。主要坑点不是 nginx 本身，而是上游对 Host 的判断、HTTPS 证书落点，以及上游返回绝对地址 Location 时把公网 IP 泄露给浏览器。最终状态是：域名解析到中间层，中间层同时服务 HTTP/HTTPS，再把请求稳定转发到博客源站。

01目标与最终链路

原始目标很直接：用户访问 hanami.top 或中间层 IP 时，拿到后端博客服务的内容。后端博客已经部署在 47.101.44.162，中间层负责对外暴露 80 和 443，并隐藏后端的直接访问细节。

02第一步：让中间层真正能代理

中间层一开始没有 nginx，80 端口也没有服务监听。确认系统是 CentOS 7 后，先通过 EPEL 安装 nginx，再新增一个反向代理配置，把所有路径转发到博客源站。

location / {
    proxy_pass http://47.101.44.162;
    proxy_http_version 1.1;
    proxy_set_header Host 47.101.44.162;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

这里最关键的一行是 proxy_set_header Host 47.101.44.162;。如果把浏览器访问的原始 Host 原样透传给源站，源站会看到 hanami.top，而它当时并不接受这个 Host，于是返回 403。

LESSON
反向代理不只是把 TCP 流量转过去。对于 HTTP 服务，Host 往往就是路由条件。中间层换 IP 通常不需要改源站，但上游接受哪个 Host 必须明确。

03第二步：把域名和 HTTPS 放到边缘

域名解析已经指向中间层后，下一步是在中间层终止 TLS。证书覆盖 hanami.top 和 www.hanami.top，私钥和证书匹配后，文件被放入 nginx 的证书目录，并将同一个代理服务扩展到 443。

listen 443 ssl http2 default_server;
server_name hanami.top www.hanami.top 206.119.160.252;

ssl_certificate     /etc/nginx/ssl/hanami.top.pem;
ssl_certificate_key /etc/nginx/ssl/hanami.top.key;
ssl_protocols TLSv1.2 TLSv1.3;

这样做以后，浏览器和中间层之间是 HTTPS；中间层到源站仍然走 http://47.101.44.162。对于这个博客场景，这个分层已经足够清晰：公网入口负责证书，源站继续专注内容服务。

04第三步：修掉泄露源站 IP 的 301

最后一个问题出现在 /blog 这个不带尾斜杠的路径。源站会把它规范化为 /blog/，但它生成的绝对跳转地址是 http://47.101.44.162/blog/。浏览器收到这个 Location 后就会直接跳去源站 IP。

修复分两层：中间层自己处理最常见的 /blog 跳转；其余从源站返回的绝对 Location，统一重写回当前访问的域名和协议。

location = /blog {
    return 301 $scheme://$host/blog/;
}

location / {
    proxy_pass http://47.101.44.162;
    proxy_redirect ~^http://47\.101\.44\.162(?::80)?(/.*)$ $scheme://$host$1;
}
检查项 结果 意义
https://hanami.top/blog 301 到域名 跳转目标是 https://hanami.top/blog/，不再暴露源站 IP。
https://hanami.top/blog/ 200 OK 最终博客页面可通过 HTTPS 正常访问。
TLS 握手 Verification OK 证书链有效，证书主体是 hanami.top。

05这次配置留下的原则

  • 中间层可以替换。 以后如果 206 这台机器换成别的 IP，只要新机器部署同样的 nginx 配置，并把 DNS 指过去，源站通常不用改。
  • Host 要显式设计。 面向用户的 Host 和面向源站的 Host 可以不是同一个。这里用户看到 hanami.top，源站看到 47.101.44.162。
  • 证书放在公网入口。 谁对浏览器提供 HTTPS，谁就持有并加载对应域名的证书。
  • Location 也属于代理面。 只处理请求头不够，响应里的绝对跳转也要检查，否则用户会被带到内部或源站地址。
