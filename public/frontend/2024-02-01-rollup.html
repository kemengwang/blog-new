02Build Hooks

构建过程中会有在各个时期执行的构建hook 我们可以改变hook从而改变构建过程

hook类型分为函数或者对象 (一般使用对象)

hook类型

(注意 是多个同类型hook 才会有顺序定义)

  • async 返回promise 不然就是sync
  • first 顺序执行 直到有返回值 而不是null or undefined.
  • sequential 多个插件实现了同一个钩子时,按照指定的顺序依次执行,一个钩子是异步的，那么后续的同类钩子会等待当前钩子解析完成后再继续执行。
  • parallel 类似于顺序执行，然而，如果一个钩子是异步的，后续的同类钩子会并行执行，不会等待当前钩子完成。

对象属性

order: "pre" | "post" | null 控制使用相同hook的plugin执行顺序 sequential: boolean parallel类型的hook开启 等待前面异步执行完毕 执行自己 后续继续异步

各种hook:link

  • buildEnd (async, parallel)构建结束 输出文件之前outputOptions 构建中的错误会传入该hook
  • buildStart (async, parallel) 构建开始 options 之后 处理传入的options
  • closeWatcher
  • load（async, first） 通过resolveId or resolveDynamicImport 拿到文件 load吐出内容，涉及ast map 摇树 meta
  • moduleParsed （async, parallel）每次有模块被完全解析时触发 （后面还有import模块时 进入resolved）
  • onLog （sync, sequential）处理日志 return false对应日志将被过滤 不然穿到下个plugin
  • options （async, sequential） 空竹options 返回null 啥都不变 只读使用buildStart
  • resolveDynamicImport (async, first) 动态解析 可以启用外部依赖 或者依然- resolveId 解析
  • resolveId (async, first) 解析import 使用
  • shouldTransformCachedM
  • transform (async, sequential) 转换
  • watchChange

注意整体build构建流程为 入口resolve 两次 =>load => transform ,入口页面所有import的文件 resolve(异步),完成后load(该页面的所有import文件) ,完成后 transform(该页面的所有import文件)，解析import文件 分析模块 继续 resolve =>load => transform (注意 此时是多个分析模块并行所以resolve =>load => transform多条 )

多个入口 意味着多个bundle ，load过程可能由于其他入口打包产出的cache也可以复用 chunk也会做到共用 writebundle 会被触发多次 但是closebundle只有一次

this.build不同 已是一个新流程了

03Output Generation Hooks

输出生成钩子 相比构建钩子(解析模块、生成代码) ，该钩子主要用于生成最终输出文件之前或之后执行自定义操作

这些hook会被bundle.generate(outputOptions) or bundle.write(outputOptions). 分别调用 bundle.generate(outputOptions): 返回一个包含生成的代码的对象 与文件系统无关 bundle.write(outputOptions)： shen cshenc
