总结业务中曾用到的一些性能优化手段

DATE · 2024-08-20 TAGS · React, Frontend SOURCE · old blog
TL;DR
总结业务中曾用到的一些性能优化手段

以下都是在函数式组件中，class不考虑

01Memo 和 useCallback

react在执行渲染时，父组件渲染一定回引发子组件渲染。虽然右diff兜底，不用担心dom重复渲染，但是在一些性能瓶颈下，我们希望组件不要被重新执行

使用Memo包裹组件，第二个参数可以传函数也可以不穿，主要比较props是否相等，使用object.is判断，一般直接包裹。由于父组件重新渲染，定义的函数重新定义就是不同的函数了，要么移出组件定义，要用useCallback包裹

02Fragment

使用Fragment减少组件不必要的组件节点

03key

绑定正确的key让diff比较时确定vdom相等

04组件颗粒度

定义组件遇到频繁setstate时将其细化到小的组件，不要影响父组件

还右像是弹窗的场景，可以将内部控制显示、隐藏、动画的其他逻辑执行封装子组件，这样未打开时dom不先行渲染，内部逻辑不会执行。这里就是渲染函数的层级了

如果这个弹窗动画复杂，优化动画的话可以先行挂载，使用opcaity

05错误边界

使用react-error-boundary库，引用ErrorBoundary组件，防止页面完全崩溃导致白屏（react有些错误会导致页面直接清空）link

也可以定义class组件 getDerviedStateFromError返回心得错误状态，展示降级ui，componentDidCatch记录错误信息

06React.lazy和React.Suspense

使用React.lazy和React.Suspense延迟加载不需要立马使用的组件

Suspense + 错误边界 可以和use结合使用，本质上就是throw抛出被外面捕获，throw 错误就边界捕获，throw promise就suspense捕获

07startTransition

startTransition(scope) scope 是一个回调函数，里面的更新任务都会被标记成过渡更新任务，过渡更新任务在渲染并发场景下，会被降级更新优先级，中断更新。

const [ isPending , startTransition ] = useTransition ()

在过度中isPending会被设置为true。

高性能机器其实反而体验不好，这个就是降低渲染的优先级

const [query, setQuery] = useState('');const deferredQuery = useDeferredValue(query);

useDeferredValue会返回延迟的值也是设置transiton，但是对值使用，返回延迟值，比startTransition更慢些

实际上上面所述，可以看成内置的防抖截流
